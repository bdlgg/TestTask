# Тестовое задание C# Junior/Middle

В данном репозитории реализовано решение **трёх задач** из тестового задания:
1. Компрессия и декомпрессия строки.
2. Потокобезопасный статический сервер для параллельного чтения и последовательной записи.
3. Консольная программа для стандартизации лог-файлов.

Постановка задач соответствует выданному тестовому заданию: компрессия строк, организация многопоточного доступа к общему счётчику и приведение логов к единому формату с вынесением невалидных строк в отдельный файл.

---

## Структура проекта

Проект состоит из следующих основных файлов:

- `Program.cs` — точка входа в приложение, меню выбора задания и запуск демонстрации каждого из трёх решений.
- `Compression.cs` — реализация алгоритмов компрессии и декомпрессии строки.
- `Server.cs` — потокобезопасный статический класс с методами `GetCount()` и `AddToCount(int value)`. Для синхронизации используется `ReaderWriterLockSlim`, что позволяет нескольким читателям читать параллельно, а запись выполнять строго эксклюзивно.
- `LogProcessor.cs` — обработка лог-файла, разбор двух входных форматов, нормализация уровней логирования, запись валидных строк в итоговый файл и невалидных — в `problems.txt`.
- `LogEntry.cs` — модель нормализованной записи лога.
- `test-logs.txt` — пример входного лог-файла для проверки 3 задания. В нём есть и валидные, и невалидные записи.
- `output.txt` — пример результата обработки валидных логов.
- `problems.txt` — пример файла с невалидными строками.

# Задание 1. Компрессия / декомпрессия строки

## Что делает решение

Реализованы два метода:

- `Compress(string str)` — сжимает строку по правилу: подряд идущие одинаковые символы заменяются на `символ + количество`.
- `Decompress(string str)` — восстанавливает исходную строку из сжатой формы.

Если символ встречается один раз подряд, число не добавляется.
Например:

```text
aaabbcccdde -> a3b2c3d2e
```

Алгоритм реализован в `Compression.cs`:
сжатие строится через проход по строке и подсчёт длины серии одинаковых символов, а распаковка — через чтение символа и следующего за ним числа любой длины.

# Задание 2. Потокобезопасный сервер

## Что делает решение

Реализован статический класс `Server` с общим целочисленным полем `count` и двумя методами:

- `GetCount()` — чтение значения.
- `AddToCount(int value)` — добавление значения к общему счётчику.

Для синхронизации используется `ReaderWriterLockSlim`:

- несколько потоков-читателей могут выполнять `GetCount()` одновременно;
- запись через `AddToCount()` выполняется только одним потоком в конкретный момент времени;
- пока идёт запись, чтение блокируется до её завершения.

Это соответствует требованию задания про параллельных читателей и последовательных писателей.

---

## Что ожидать при проверке задания 2

В консоли будет примерно такой вывод:

```text

Читатель 2: count = 0
Читатель 3: count = 0
Писатель 1: добавил 10
Писатель 1 закончил
Писатель 2: добавил 20
Писатель 2 закончил
Писатель 3: добавил 30
Писатель 3 закончил

Читатель 5: count = 60
Читатель 6: count = 60
Итоговое значение count: 60
```

Порядок строк может немного отличаться из-за многопоточности, что нормально.
Главное, что:

- значение не повреждается при параллельной работе;
- запись не выполняется одновременно несколькими потоками;
- итоговая сумма корректна.

Так как в демонстрации три писателя добавляют `10`, `20` и `30`, итоговое значение должно быть **60**.

# Задание 3. Стандартизация лог-файла

## Что делает решение

Решение обрабатывает входной лог-файл построчно и поддерживает **два входных формата**, указанных в задании. Если строка валидна, она преобразуется в единый табличный формат и записывается в выходной файл. Если строка невалидна, она записывается в `problems.txt` без изменений.

Обработка реализована в `LogProcessor.cs`. Основная логика такая:

- `ProcessFile(...)` читает входной файл и распределяет строки между `output.txt` и `problems.txt`.
- `TryParseLine(...)` определяет формат строки: если есть символ `|`, используется формат 2, иначе формат 1.
- `TryParseFormat1(...)` обрабатывает строки вида  
  `10.03.2025 15:14:49.523 INFORMATION Сообщение`.
- `TryParseFormat2(...)` обрабатывает строки вида  
  `2025-03-10 15:14:51.5882| INFO|11|Method.Name| Сообщение`.
- `NormalizeLevel(...)` приводит уровни логирования к стандартным значениям:
    - `INFORMATION` → `INFO`
    - `INFO` → `INFO`
    - `WARNING` → `WARN`
    - `WARN` → `WARN`
    - `ERROR` → `ERROR`
    - `DEBUG` → `DEBUG`

Если поле метода отсутствует или пустое, в результат подставляется `DEFAULT`.

# Краткое описание логики решений

## Задание 1
Используется классический подход со счётчиком повторов:
- при компрессии подряд идущие одинаковые символы объединяются;
- при декомпрессии символ повторяется указанное число раз.

## Задание 2
Используется `ReaderWriterLockSlim`, потому что он хорошо подходит под условие:
- чтений много;
- запись должна быть эксклюзивной;
- читатели не должны мешать друг другу.

## Задание 3
Каждая строка анализируется независимо:
- определяется формат;
- валидируются дата и уровень логирования;
- собирается объект `LogEntry`;
- формируется итоговая строка через табуляцию;
- при ошибке строка уходит в `problems.txt`.
